## 0. プロダクト名とタグライン

**Foundry Brain** — Company Brain, built for the fab.

### 設計思想: 1つのブレイン + 複数スキル（GStack 型）

Garry Tan の [gstack](https://github.com/garrytan/gstack) が「1つの Claude Code を CEO / Designer / QA... という役割スキルの束にする」のと同じ発想を、プロダクトのアーキテクチャに採る。**プロダクトの実体は Claude Code のスキル群**（`.claude/skills/`）。

- **ブレイン** = 共通のデータ基盤（MES / FDC / Metrology の3系統）+ 蓄積メモリ
- **スキル** = 熟練歩留まりエンジニアの暗黙知を切り出した個々の能力。`SKILL.md`（Markdown）として定義し、同じデータ基盤・メモリの上に載る。後から増やせる
- 効果: 「点のツール」ではなく「増殖するプラットフォーム」として語れる。excursion を解くたびにブレインのメモリが増え、同じデータ基盤に新スキルが積み上がる

#### ソースの構成（`.claude/skills/`）

| パス | 中身 |
| --- | --- |
| `.claude/skills/gstack/` | Garry Tan の gstack 実リポを同梱（54スキル + lib/bin 等）。Foundry Brain が乗る土台 |
| `.claude/skills/<skill>` ×54 | gstack 各スキルへの**シンボリックリンク**（`→ gstack/<skill>`） |
| `.claude/skills/foundry-brain/` | **ブレイン** — 共通データ基盤（`data/` 3系統CSV）+ gbrain 記憶ツール（`bin/`） |
| `.claude/skills/ai-yield-engineer/` | **スキル#1** — 故障分析・Hold推奨・UI起動 |
| `.claude/skills/hold-or-ship/` 等 | **将来スキル**（スタブ）— 同じ `foundry-brain/data/` を参照 |
| `.claude/skills/excursion-diagnosis/` | 旧名 → `/ai-yield-engineer` へリダイレクト |

**スキル#1「Excursion Diagnosis」の挙動**: `ai-yield-engineer/SKILL.md` の調査手順に従い、**`foundry-brain/data/`** の3系統CSVを query → 不良ロット特定 → 装置ルート追跡 → センサードリフト検出 → 原因確定＋Hold推奨。**その上で `visualizer/` を起動**し、同じ調査をブラウザで再生。

- **ハッカソンで前面に出すのはスキル#1のみ**。他スキル（Hold-or-Ship / Drift Watch / Commonality）は強く売り込まず、「データが溜まるのでこういう使い方もできる」という**軽い展望**として最後に触れるだけ（スコープを広げない）

## 1. 課題設定

### 解くべき課題
半導体工場（ファブ）では、突然「良品率（歩留まり／yield）」が落ちる異常（excursion）が起きる（例: 90% → 78%）。現場が知りたいのは2つ:

1. **原因はどの装置・どの工程か？**
2. **今ラインを流れている製品を、止めるべきか・出荷していいか？**

この判断がいまだに手作業。担当は「歩留まりエンジニア（yield engineer）」という専門職（米国で年収$100k〜$140k）。異常直後の「止めるか流すか」の判断待ち時間が最大のボトルネック。

### なぜ解けていないか

データは1日テラバイト単位で生まれている。問題は3つのシステム（装置センサー/製造記録/検査結果）がそれぞれ別のID体系・時間粒度で動いていて、突き合わせに人間の判断が必要なこと。

**「統合DBを作ればいいのでは？」が通用しない理由:**
- マッピングが静的でない: 新製品・レシピ変更のたびにJOIN条件が変わる
- ロットが途中で分割・合流・やり直しされるので単純なID追跡が効かない
- 500〜1000工程 × 数百台 × 数千パラメータの組み合わせ爆発
- 原因がしばしば「閾値内の微妙なドリフト」でアラームが鳴らない

既存ツール（KLA、Applied Materials等）はデータの可視化・相関分析まではやれるが、「どのデータをどう突き合わせるか」の最後の判断は人間の暗黙知に依存している。

**なぜAI化されてこなかったか:**
- 歩留まりデータは企業の最高機密。外部ベンダーにも見せられない
- 従来のMLは過去パターンの学習だが、excursionの原因は毎回異なり、パターン学習が的外れ
- 結果、少数の熟練エンジニアに属人化。退職で消え、新工場にはゼロから

**なぜ今解けるか:** LLMエージェントは過去パターンの学習ではなく「今あるデータを都度読んで仮説→検証する」推論。必要なのは調査手順の知識であり、プロンプトとツール定義で注入できる。

## 2. ソリューション

**スキル#1「Excursion Diagnosis」= AI歩留まりエンジニア** = 熟練エンジニアの調査プロセスを再現するAIエージェント。Foundry Brain（ファブのブレイン）が最初に備えるスキル。

「不良が増えた、原因を調べて」と話しかけると、3つのデータソースを自動で渡り歩き、「原因はこの装置」「この製品を止めろ」を根拠つきで返す。

**既存ツールとの差:**
- 既存: 相関・パターンを出す（Analyze） → 解釈は人間
- 我々: なぜ起きたか推論し（Diagnose）、隔離/出荷の判断まで出す（Act）

**インパクト:** 異常時の本当のコストは不良品より「不確実性」（仕掛品の宙吊り、エンジニアの時間、装置の空き、納期遅延）。この判断待ち時間を数時間→数分に短縮する。

## 3. 市場性

### 市場規模（フェルミ推定: 歩留まり損失からの逆算）

既存の歩留まり管理ソフト市場（~$20億）は「可視化ツール」の値札。我々が削るのはその先にある**歩留まり損失そのもの**。

**コストプール（我々が削減する損失の総額）:**
- 世界の半導体売上: 約$6,000億/年
- 歩留まり起因の損失（スクラップ・リワーク・機会損失）: 売上の5〜10% → **$300億〜$600億/年**
- これが我々のプロダクトが食い込む「痛みの総量」

**ファブ単位のROI:**
- 先端ウェーハ1枚: $5,000〜$17,000。1回のexcursionで$1M〜$10M規模の損失
- ファブあたり年間5〜15件 → **年$10M〜$50Mの歩留まり損失**
- 判断待ち時間を短縮し損失の10〜20%を防ぐだけで、**1ファブあたり年$1M〜$10Mの価値創出**

**TAM:**
- 世界のファブ: 約1,200拠点（うち先端200、レガシー含め全体が対象になりうる）
- ファブあたりのソフト支出余力: 生み出す価値の10〜20%を課金 → $200k〜$2M/ファブ
- **TAM: $1B〜$2B**（既存ソフト市場の「置き換え」ではなく、歩留まり損失$300億の削減という新しい価値層の一部を取る）

### ターゲット
- **新興ファウンドリ（最初の楔）**: 既存ツールへのロックインがなく、暗黙知もゼロ。「最初から導入」が入りやすい
- **日本: Rapidus**が政府支援2兆円超で2nmファブ建設中（2027年量産予定）。歩留まりデータは安全保障に関わる機密で、「日本のプレイヤーが国内で扱う」座組が強み
- **日米2拠点**: 両国とも新工場が今まさに立ち上がるタイミング

### 競合
- **既存ツール**（KLA Klarity、Applied Materials SmartFactory、yieldWerx、Spotfire、Galaxy Semiconductor）: 導入に数ヶ月の重厚システム。相関の可視化まで。解釈は人間任せ
- **最も近い新興: ThirdAI Automation**（2024年設立、2026年シード$3M）。ただし彼らは「装置トラブルの診断」、我々は「製品の隔離判断」。軸足が違う
- 競合の存在 = 市場が実在する証明。ピッチでは「知った上で別のangleを取っている」と見せる

## 4. デモ

### コンセプト
AIが調査する過程をリアルタイムで見せる1画面Webアプリ。答えがポンと出るのではなく、**調査の過程を"わざと見せる"**のが肝。

### 画面構成（3ペイン）
- **左**: 自然言語入力（「歩留まりが12%落ちた。原因は？」）
- **中央**: 調査ログ（データソースへの問い合わせが順次表示される）
- **右**: 原因装置の絞り込みボード（確信度バーが動き、5台→1台に収束）
- **下**: 判定カード（原因装置＋根拠＋隔離推奨＋指示書生成ボタン）

### モックデータ
実データは機密で入手不可 → モックデータに**解くべき構造を仕込む**:
- 不良ロットは全て、ある時間帯にEtch装置#3・チャンバーCを通過
- その時間帯だけRF電源がドリフト（閾値内なのでアラームなし）

→ 「データは合成だが、問題の構造は本物と同じ」と言い切れる。

### 実装メモ
- ツール: `query_fdc` / `get_lot_genealogy` / `query_metrology` / `compute_commonality`
- 保険: ライブLLMが暴れた場合のスクリプト再生モード

## 6. 5分ピッチ構成

1. **つかみ＋課題【1分】**: 歩留まりが突然落ちる。知りたいのは「どの装置か」「止めるか流すか」。今も$100k超のエンジニアが手作業
2. **なぜ解けないか【1分】**: データは3システムにバラバラ。ID・時間軸が違い単純JOINが効かない。機密で外注不可、属人化
3. **ソリューション【1分】**: AIが3データを渡り歩いて原因特定＋隔離判断。既存は相関止まり、我々は診断＋次の一手
4. **なぜ今・なぜ勝てる【1分】**: 半導体投資の9割はチップ側、運用ソフトは空白。日本は新工場＋安全保障シナジー
5. **デモ＋展望＋締め【1分】**: アラート→調査→絞り込み→判定→指示書生成。軽く展望: **「今日出すのはスキル#1（Excursion Diagnosis）。解くたびにブレインにメモリが溜まり、同じデータ基盤に次のスキル（Hold-or-Ship／Drift Watch／Commonality…）が積み上がる」**（強く売り込まず一言）。締め: **「できない理由を3つ挙げられます — 機密・複雑・属人化。その3つこそが、巨大なビジネスになる理由です」**
